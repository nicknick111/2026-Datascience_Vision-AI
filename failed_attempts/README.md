## 1. 프로젝트 배경 및 이 폴더의 존재 이유

### 무엇을 만들려 했는가

본 프로젝트는 **드론 부품(C4 커넥터, 통신 모듈, 전원 공급 장치) 표면 결함 검출 Vision AI 시스템** 개발을 목표로 합니다.  
카메라 `RPC-20F` (해상도 1280×720)로 컨베이어 위의 부품을 실시간 촬영하여,  
`['Defect', 'Normal_C4', 'Normal_Comm', 'Normal_Supply']` 4개 클래스로 판정하고  
그 결과를 PLC(공장 자동화 제어기) 및 외부 서버로 전송합니다.

### 왜 이 폴더가 존재하는가

> "좋은 아키텍처는 하늘에서 떨어지지 않는다. 수많은 '이건 왜 안 되지?'의 답을 찾아가는 과정에서 나온다."

`failed_attempts/` 폴더는 **단일 YOLOv8 모델로 탐지·각도·분류를 동시에 해결하려 했던 모든 시도**를 보존합니다.  
이 코드들은 "실패작"이 아닙니다. 각각의 파일은 다음 질문에 대한 실험 결과입니다:

- 비디오에서 프레임을 어떻게 추출하면 가장 다양한 학습 데이터가 나올까?
- 카메라 렌즈 왜곡을 어떻게 보정할까?
- 3D 환경 시뮬레이션으로 데이터를 증강할 수 있을까?
- YOLO 하나로 탐지와 분류를 동시에 하면 어떤 문제가 생길까?
- FastAPI 서버가 AI 추론 도중 타임아웃이 걸리는 이유는 무엇인가?

## 2. 최초 설계 목표 — 단일 YOLO 모델의 야심찬 도전

### 초기 아이디어

프로젝트 초기에는 "모델 하나로 모든 것을 해결하자"는 단순한 설계 목표를 세웠습니다.

```
[카메라] → [단일 YOLOv8 모델] → 위치 탐지 + 각도 계산 + 결함 분류 동시 처리 → [판정 결과]
```

이 접근법은 구조가 단순하고 추론 속도가 빠르다는 장점이 있었습니다.  
`camera_monitor_yolo.py`, `First Integrated Vision AI Inspection.py` 등이 이 아이디어의 구현 시도입니다.

### 초기 설계의 핵심 가정

| 가정 | 실제 결과 |
|------|-----------|
| YOLO 하나로 위치+분류 동시 처리 가능 | 회전된 부품에서 분류 정확도 급락 |
| 일반 BBox (Bounding Box)로 충분 | 비스듬한 부품에서 배경 노이즈 혼입 심각 |
| 단일 파일에 카메라+AI+DB+API 통합 | 코드 유지보수 불가능한 스파게티 구조 발생 |
| 3D 증강으로 데이터 부족 해결 가능 | 실제 결함 질감과 시뮬레이션 질감 간 도메인 갭 발생 |

## 4. 시계열 시행착오 흐름

```
Phase 1: 기초 확립 (카메라 + 데이터 수집)
    ↓
Phase 2: 단일 YOLO 통합 첫 시도 (카메라 모니터링 → YOLO 연동)
    ↓
Phase 3: 대규모 통합 시도 (단일 파일에 모든 기능 집약)
    ↓
Phase 4: 3D 증강 실험 (데이터 부족 해결 시도)
    ↓
Phase 5: 디버깅 및 구조화 재시도
    ↓
[한계 인식] → Two-Stage Pipeline 설계로 전환
```

### Phase 1 — 기초 확립: 카메라와 데이터

`camera_test.py`에서 카메라 연결 자체를 검증하는 것부터 시작했습니다.  
`RPC-20F` 카메라의 1280×720 해상도 출력을 확인한 뒤,  
`Intrinsic Calibration.py`로 체커보드 패턴을 촬영하여 렌즈 왜곡 보정 행렬(MTX, DIST)을 계산했습니다.

> **[교훈]** 이 단계에서 측정한 `MTX = [[922.75...], ...]`, `DIST = [[-0.052..., -0.450...]]` 값은  
> 이후 `vision_ai_main.py`의 `cv2.initUndistortRectifyMap()`에 그대로 계승됩니다.  
> 기초 측정이 얼마나 중요한지를 증명하는 사례입니다.

### Phase 2 — 카메라 모니터링에서 YOLO 연동으로

**문제 발견:**  
`camera_monitor.py`는 단순 영상 출력만 했습니다. 여기에 YOLO를 붙이려 `camera_monitor_yolo.py`를 작성했습니다.

**원인 분석:**  
일반 YOLO BBox (Axis-Aligned Bounding Box: 축에 정렬된 사각형 경계 상자) 방식으로는  
비스듬히 놓인 부품을 감쌀 때 주변 배경이 대량 혼입됩니다.  
드론 부품은 컨베이어 위에 임의 각도로 놓이기 때문에 이 문제가 치명적이었습니다.

**개선 방향:**  
`camera_monitor(2026-03-23 수정).py`에서 탐지 후 각도 계산 로직을 추가해봤지만,  
YOLO의 기본 탐지기가 각도 정보를 제공하지 않아 별도 처리가 필요함을 인식했습니다.

### Phase 3 — 대규모 통합: 첫 번째 "모든 것을 하나에"

`First Integrated Vision AI Inspection.py` (15KB)는 가장 야심찬 시도였습니다.

**문제 발견:**  
HTTP API 전송이 메인 루프를 블로킹(Blocking: 다른 작업을 막고 혼자 점유)하여  
카메라 프레임 드랍과 PLC 응답 지연이 발생했습니다.

**원인 분석:**  
I/O 바운드 작업(네트워크 전송, 디스크 저장)과 CPU 바운드 작업(AI 추론)을  
하나의 스레드에서 동기 처리하면 서로가 서로를 막는 병목 현상이 발생합니다.

**개선 방향:**  
`unified_vision_system.py` (12KB)에서 기능을 함수로 분리하고  
`threading` 모듈로 I/O 작업을 별도 스레드로 내보내려 시도했습니다.

### Phase 4 — 3D 증강 실험: 데이터 부족 문제 해결 시도

데이터 수집 한계를 돌파하기 위해 `Test_AI_Model_Train.py`에서  
**3D 환경 시뮬레이션 기반 데이터 증강**을 시도했습니다.

이 접근법은 3개 버전으로 발전했습니다:
- V1: `Test_AI_Model_Train.py` — 기본 3D 증강
- V2: `3D 환경 증강 학습기 (Split 구조 대응 및 최적화).py` — train/val 분리 추가
- V3: `3D 환경 증강 학습기 (상대 경로 & 원본 보호).txt` — 버그 수정

**문제 발견:**  
3D 렌더링 이미지와 실제 카메라 촬영 이미지 사이에 **도메인 갭** (Domain Gap: 학습 데이터와 실제 데이터의 분포 차이)이 발생했습니다. 실제 결함의 질감(스크래치, 이물질)을 시뮬레이션으로 재현하는 것이 한계에 부딪혔습니다.

**개선 방향:**  
시뮬레이션 증강 대신 **실제 카메라로 수집한 데이터 + OpenCV 기반 증강** (밝기/대비/회전 조합)으로 전략 전환.  
이것이 최종적으로 `balance_dataset.py`의 7가지 증강 기법으로 이어집니다.

## 5. 디버깅 흔적이 말해주는 기술적 어려움

### fix_bug.py / fix_model.py — 모델 가중치 불일치의 벽

`fix_bug.py`와 `fix_model.py`의 존재는 YOLOv8 학습 환경과 추론 환경의 Python 버전,  
Ultralytics 버전, 모델 구조가 달라지면 `state_dict` (모델 가중치 사전)의 키 이름이 바뀌어  
로드가 실패함을 의미합니다. 이를 수동으로 리매핑하는 스크립트가 필요했다는 것은  
버전 관리와 환경 일관성의 중요성을 온몸으로 깨달은 증거입니다.

### cache_delete.py — YOLO 캐시 오염 문제

YOLO는 학습 시 데이터셋을 `.cache` 파일로 저장합니다.  
데이터셋 구조가 변경되면 구 캐시를 그대로 참조하여 엉뚱한 데이터로 학습되는 문제가 발생합니다.  
`cache_delete.py`는 이 캐시를 강제 삭제하는 단 한 가지 목적의 유틸리티로,  
"이 문제를 여러 번 겪었다"는 것을 증명합니다.

### pt_to_onnx.py — 경량화 시도의 어려움

ONNX (Open Neural Network Exchange: 다양한 프레임워크 간 모델 교환 포맷)로  
변환을 시도한 것은 엣지 디바이스 배포를 고려한 것이었습니다.  
그러나 단일 YOLO 모델의 분류 정확도 자체가 해결되지 않은 상태에서  
경량화는 의미가 없음을 깨닫고 이 방향은 일시 보류했습니다.

## 6. 핵심 교훈 — 왜 단일 모델이 한계에 부딪혔는가

| 도전 과제 | 단일 YOLO 방식의 한계 | Two-Stage 해결책 |
|-----------|----------------------|-----------------|
| 임의 각도 부품 탐지 | 일반 BBox가 배경 노이즈 포함 | OBB(회전 경계 상자)로 정밀 탐지 |
| 정밀 결함 분류 | YOLO 분류 헤드는 거친 특징 추출 | EfficientNet-B0 전용 분류기 사용 |
| 모델 구조 복잡도 | 탐지+분류 동시 최적화 충돌 | 역할 분리로 각각 최적화 |
| 실시간 API 통신 | 블로킹 루프로 타임아웃 발생 | Non-blocking 트리거 + Polling 구조 |
| 데이터 품질 | 3D 시뮬레이션 도메인 갭 | 실제 촬영 + OpenCV 증강 전략 |

단일 모델이 "모든 것을 잘 하게" 만들려면 학습 데이터도 "모든 상황을 커버"해야 합니다.  
그러나 탐지 정확도와 분류 정확도의 최적화 방향은 서로 다른 목적 함수(Loss Function)를 가집니다.  
이 둘을 하나의 모델에서 동시에 최적화하는 것은 근본적인 구조적 한계였습니다.

## 7. Two-Stage Pipeline으로의 연결

`failed_attempts/` 폴더에서 얻은 모든 교훈은 `two_stage_pipeline/`의 설계 원칙으로 직결됩니다.  
`Intrinsic Calibration.py`에서 측정한 카메라 행렬은 `vision_ai_main.py`에 그대로 살아있고,  
`unified_vision_system.py`에서 고민하던 스레드 분리 아이디어는 `api_server.py`의 비동기 구조로 완성되었으며,  
3D 증강의 실패는 `balance_dataset.py`의 7가지 실사 기반 증강 전략으로 진화했습니다.  
이 폴더는 "실패의 무덤"이 아니라 **최종 아키텍처를 위한 지식 은행**입니다.
